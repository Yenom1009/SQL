# 人事系统需求分析

## 一、系统目标

本系统基于 MySQL 数据库和 Flask 框架，旨在实现企业人事信息的数字化管理，满足不同角色用户对员工信息的查询、维护及权限控制需求。通过结构化存储和流程化管理，提升人事管理效率，确保数据安全与业务逻辑合规性。

## 二、用户角色与权限需求

### 1. **角色划分**

- **员工**：系统基础用户，仅能访问个人相关信息。
- **领导**：具备管理权限的用户，可操作全量数据并执行维护功能。

### 2. **权限细则**（员工有的权限领导一定有）

| **功能模块**        | **员工权限**                         | **领导权限**                 |
| ------------------- | ------------------------------------ | ---------------------------- |
| 个人信息查看        | ✅ 查看姓名、岗位、薪资等             | ✅ 查看所有员工信息           |
| 考勤 / 请假记录查看 | ✅ 可以考勤、写请假申请、查看本人记录 | ✅ 查看所有员工记录           |
| 员工信息修改        | ❌ 无权限                             | ✅ 修改岗位、薪资（需合规）   |
| 岗位变动管理        | ❌ 无权限                             | ✅ 新增 / 修改变动记录        |
| 部门 / 岗位配置     | ❌ 无权限                             | ✅ 维护部门结构与岗位薪资范围 |

## 三、业务功能需求

### 1. **员工信息管理**

- 核心功能：
  - 员工档案存储（编号、姓名、学历、联系方式等）。
  - 学历字段强制约束（仅允许 “中专、高中、大专、本科、硕士、博士”）。
  - 岗位编号与薪资关联，通过外键引用岗位表薪资范围。
- 操作场景：
  - 新员工入职：分配 “待定” 岗位（属于 “空” 部门），待审批后正式定岗。
  - 员工离职：先将部门调整为 “空”，再删除员工信息（需确保无考勤 / 请假记录）。

### 2. **部门与岗位管理**

- 部门管理：
  - 唯一部门名称约束（如 “技术部”“市场部”），支持 “空” 部门用于临时状态。
  - 部门负责人需为在职员工（通过员工编号外键关联），负责人变动需先在部门表更新。
- 岗位管理：
  - 每个岗位绑定所属部门，设置薪资上下限（如 “技术岗” 最低工资 8000 元，最高工资 15000 元）。
  - “待定” 岗位固定属于 “空” 部门，用于员工入职 / 离职过渡期。

### 3. **变动与考勤管理**

- 岗位变动：
  - 记录变动前后的岗位、薪资，自动校验新薪资是否在目标岗位范围内。
  - 若员工为部门负责人，禁止直接修改岗位（需先解除负责人身份）。
- 考勤与请假：
  - 考勤记录唯一标识员工每日出勤情况（通过日期 + 员工编号唯一约束）。
  - 请假申请需关联员工编号，领导审批后生成记录，自动校验日期合法性（结束日期≥开始日期）。

### 4. **系统权限管理**

- 登录验证：
  - 基于用户表（user_id、password、role）验证身份，密码加密存储（如 bcrypt）。
  - **登录后根据角色（员工 / 领导）动态渲染前端界面（如隐藏 “修改” 按钮**）。
- 数据隔离：
  - 员工端通过 user_id 关联员工编号，强制限制查询范围为本人记录。
  - 领导端可访问全量数据，但操作需遵循业务规则（如薪资修改需符合岗位范围）。

## 四、数据需求与约束

### 1. **数据实体关系**

- **员工表** ↔ **岗位表**：通过 “岗位编号” 外键关联，员工薪资需在岗位薪资范围内。
- **部门表** ↔ **岗位表**：岗位通过 “部门编号” 归属部门，“空” 部门对应 “待定” 岗位。
- **员工表** ↔ **系统用户表**：通过 user_id 一对一关联，确保账号与员工唯一绑定。

### 2. **完整性约束**

- **实体完整性**：员工编号、部门编号、岗位编号为主键，禁止重复。
- **参照完整性**：外键字段（如岗位编号、部门负责人）需引用存在的主键值。
- 用户定义完整性：
  - 学历、性别字段取值固定（如性别仅 “男 / 女”）。
  - 岗位薪资范围需满足 “最低工资≤最高工资”。
  - 请假记录日期需合法（结束日期不早于开始日期）。

3. **岗位表**：<u>岗位编号</u>，岗位名称，部门编号（外键），最低工资，最高工资

   说明：

   岗位编号是主键；

   所属部门编号是外键依赖；

   每个岗位都有对应的最低和最高工资限制（有效防止乱改工资）；

   存在一个岗位名为“待定”，属于部门“空”，用于刚入职或待离职时的人事变动；

4. **员工岗位变动表**：<u>变动记录编号</u>，员工编号（外键），变动时间，变动前岗位编号（外键），变动后岗位编号（外键），变动前薪资，变动后薪资，

   说明：

   变动记录编号是主键；

   变动前岗位编号（外键），变动后岗位编号（外键）是外键；

   进入修改系统后，同时修改岗位，薪资（如果不变就填和原来一样的就好了），填完之后检查，如果新的薪资不在新岗位的min-max中，拒绝修改；如果该员工是原本部门的负责人，拒绝修改。（**员工的岗位编号可以检查出部门编号，检查该部门负责人编号是否和该员工一致，一致则不允许修改，也就是说，要想改动部门负责人，需要先在部门表进行修改**）

5. **考勤表**：<u>考勤记录编号</u>，员工编号（外键），日期

   说明：

   考勤记录编号是主键（每新增一条考勤记录，自动新生成记录编号）；

   员工编号是外键；

   

6. **请假记录表**：

   | **字段**            | **类型 / 约束** | **说明**                                     |
   | ------------------- | --------------- | -------------------------------------------- |
   | <u>请假申请编号</u> | INT（主键）     | 唯一标识请假记录，自增主键                   |
   | 员工编号            | INT（外键）     | 关联员工表（Employee. 员工编号），标识请假人 |
   | 请假类型            | VARCHAR(50)     | 如年假、事假、病假等                         |
   | 开始日期            | DATE            | 请假起始日期                                 |
   | 结束日期            | DATE            | 结束日期                                     |
   | 申请时间            | DATETIME        | 系统自动记录申请时间                         |

   说明：

   请假申请编号是主健；

   员工成功向他的领导申请请假成功才能新添一条记录（至于他线下线上和领导怎么斡旋的，本系统不关心，如果领导不批假，直接不生成请假记录）；

9. **系统用户表**：<u>user_id</u>，password，role  

   

   说明：

   1. **验证登录信息**：后端接收到登录信息后，在用户表（User）中查询是否存在匹配的 username 和 password 记录。如果不存在，返回错误提示，拒绝登录；如果存在，继续下一步。
   
   2. **获取角色权限**：根据登录成功的 user_id，从用户表中获取对应的 role（角色权限）。
